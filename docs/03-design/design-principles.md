# Design Principles

## Product design direction
This repo should feel like an operating system, not a marketing site and not a random document dump.

## Principles
- Clarity over decoration
- Consistency over novelty
- Honest states over fake polish
- Guidance over noise
- Durable structure over one-session cleverness

## Practical implications
- README should be the clean front door.
- Folder names should explain the workflow without extra explanation.
- The repo should be readable in plain markdown on GitHub.
- Examples should teach usage, not just exist as placeholders.
