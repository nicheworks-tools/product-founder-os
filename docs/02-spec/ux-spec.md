# UX Spec

## Main screens / states
This repo is doc-driven, not app-screen-driven.
Its practical UX is the experience of reading and using the repo structure.

## Primary actions
- understand what the repo is for
- ask / answer what is being built
- read the current product state
- update definition and planning docs
- continue execution from current state

## Empty state expectations
A fresh repo should still clearly communicate how to start.

## Error state expectations
If the repo is drifting, the docs should make that visible rather than hiding it.
